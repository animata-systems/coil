# COIL:ru-standard — Стандартный русский диалект

> «Пиши сценарий на языке, на котором думаешь.»

**Статус:** официальный диалект, часть стандартной поставки.

Этот диалект — каноническая русскоязычная форма COIL. Все ключевые слова совпадают с теми, которые используются в спецификации языка. Если вы читаете спеку — вы уже знаете этот диалект.

---

## Принципы дизайна

1. **Повелительное наклонение.** Операторы — глаголы в форме императива: ОПРЕДЕЛИ, ПОЛУЧИ, ДУМАЙ. Протокол обращается к среде исполнения как к исполнителю: «сделай это».

2. **Семантическая точность.** Ключевое слово отражает поведение оператора, а не его техническую реализацию. `НАПИШИ` — потому что пользователь «пишет в канал», а не «отправляет пакет».

3. **Одно слово, где возможно.** Многословные формы допускаются только там, где одно слово было бы двусмысленным: `ОТВЕТ НА`, `НЕ БОЛЕЕ`.

4. **Категориальная подсказка.** Коннотация слова намекает на категорию оператора: блокирующий (`ПОЛУЧИ`, `ЖДИ`), запускающий (`ДУМАЙ`, `ВЫПОЛНИ`, `НАПИШИ`), мгновенный (`ОПРЕДЕЛИ`, `УСТАНОВИ`).

## Операторы Core

| ID | RU | Семантика |
|---|---|---|
| `Op.Actors` | `УЧАСТНИКИ` | Объявление участников протокола. Те, кто действует в сценарии. |
| `Op.Tools` | `ИНСТРУМЕНТЫ` | Объявление доступных инструментов. Чем агенты будут работать. |
| `Op.Define` | `ОПРЕДЕЛИ` | Создание нового именованного значения. Как в математике: определить термин. |
| `Op.Set` | `УСТАНОВИ` | Изменение существующего значения. Стандартный глагол присваивания. |
| `Op.Receive` | `ПОЛУЧИ` | Блокирующая привязка значения из среды. Протокол ждёт, пока среда предоставит значение — не забирает его активно. |
| `Op.Think` | `ДУМАЙ` | Когнитивный шаг LLM. Модель получает постановку задачи и думает над ней. |
| `Op.Execute` | `ВЫПОЛНИ` | Вызов инструмента. Покрывает как одноразовые вызовы, так и интерактивные сессии. |
| `Op.Send` | `НАПИШИ` | Отправка сообщения в канал. Не «отправь» — пользовательский фрейм: «напиши в чат», «напиши эксперту». |
| `Op.Wait` | `ЖДИ` | Точка синхронизации. Приостановить выполнение, пока обещания не разрешатся. |
| `Op.Exit` | `ВЫХОД` | Завершение протокола. Одна строка, без аргументов. |
| `Kw.End` | `КОНЕЦ` | Закрытие блока. Каждый блочный оператор открывается ключевым словом и закрывается `КОНЕЦ`. |

## Модификаторы

### Оснащение (ДУМАЙ)

| ID | RU | Семантика |
|---|---|---|
| `Mod.Via` | `ЧЕРЕЗ` | Какую модель LLM использовать. `ЧЕРЕЗ $fast_model` — направить задачу через эту модель. |
| `Mod.As` | `КАК` | Квалификация решателя (скиллы). `КАК $аналитик` — думай как этот эксперт. Только ссылки. |
| `Mod.Using` | `ИСПОЛЬЗУЯ` | Доступные инструменты для LLM. `ИСПОЛЬЗУЯ !поиск, !калькулятор` — модель может вызвать эти инструменты. |

### Постановка (ДУМАЙ)

| ID | RU | Семантика |
|---|---|---|
| `Mod.Goal` | `ЦЕЛЬ` | Зачем решаем. Что хотим получить в итоге. |
| `Mod.Input` | `ВХОД` | Условие задачи. Данные, с которыми работать. |
| `Mod.Context` | `КОНТЕКСТ` | Дополнительные данные, фон, ограничения. |
| `Mod.Result` | `РЕЗУЛЬТАТ` | Что LLM должна определить. Структурный микросинтаксис. |

### Адресация (НАПИШИ)

| ID | RU | Семантика |
|---|---|---|
| `Mod.To` | `КУДА` | Адрес канала. `КУДА #поддержка` — доставить в этот канал. |
| `Mod.For` | `КОМУ` | Получатель. `КОМУ @эксперт` — это сообщение для этого участника. |
| `Mod.ReplyTo` | `ОТВЕТ НА` | Ссылка на сообщение, на которое отвечаем. |
| `Mod.Await` | `ЖДАТЬ` | Политика ожидания ответа. `ЖДАТЬ ЛЮБОЙ` — ждать хотя бы один ответ. |
| `Mod.Timeout` | `НЕ БОЛЕЕ` | Таймаут. `НЕ БОЛЕЕ 10м` — отказаться через 10 минут. |

### Синхронизация (ЖДИ)

| ID | RU | Семантика |
|---|---|---|
| `Mod.On` | `НА` | Ожидаемые обещания. `НА ?план, ?данные` — это обещания, которых ждём. |
| `Mod.Mode` | `РЕЖИМ` | Режим ожидания. Сколько обещаний должно разрешиться. |
| `Pol.All` | `ВСЕ` | Ждать все перечисленные обещания. |
| `Pol.Any` | `ЛЮБОЙ` | Ждать любое одно из перечисленных. |
| `Pol.None` | `НЕТ` | Не ждать ответа. Отправил и забыл. |
| `Mod.Timeout` | `НЕ БОЛЕЕ` | Таймаут. То же ключевое слово, что в НАПИШИ — единообразие. |

### Вызов инструмента (ВЫПОЛНИ)

| ID | RU | Семантика |
|---|---|---|
| `Mod.Using` | `ИСПОЛЬЗУЯ` | Какой инструмент вызвать. `ИСПОЛЬЗУЯ !поиск` — обязателен, ровно один. |

## Extended-операторы

| `Op.Repeat` | `ПОВТОРЯЙ` | Цикл с обязательным лимитом. `ПОВТОРЯЙ ДО $готово НЕ БОЛЕЕ 5`. |
| `Op.Each` | `КАЖДЫЙ` | Итерация по элементам списка. `КАЖДЫЙ $задача ИЗ $план.файлы`. |

| `Op.Signal` | `СИГНАЛ` | Отправка данных в существующий поток. |

### Модификаторы итерации

| ID | RU | Семантика |
|---|---|---|
| `Mod.Until` | `ДО` | Условие выхода из цикла. `ПОВТОРЯЙ ДО $готово НЕ БОЛЕЕ 5`. |
| `Mod.Limit` | `НЕ БОЛЕЕ` | Лимит итераций (в ПОВТОРЯЙ). Обязателен — цикл без лимита невалиден. |
| `Mod.From` | `ИЗ` | Источник списка для итерации. `КАЖДЫЙ $элемент ИЗ $список`. |

## Сигилы ссылок

Сигилы — часть лексического ядра и не меняются между диалектами.

| Сигил | Значение |
|---|---|
| `$` | Значение |
| `?` | Обещание |
| `@` | Участник |
| `!` | Инструмент |
| `#` | Канал |
| `~` | Поток |

## Контекстное разрешение

Фраза `НЕ БОЛЕЕ` маппится на два абстрактных идентификатора. Контекст оператора однозначно определяет, какой из них имеется в виду:

| Фраза | Контекст | Абстрактный ID |
|---|---|---|
| `НЕ БОЛЕЕ` | Внутри `НАПИШИ` или `ЖДИ` | `Mod.Timeout` |
| `НЕ БОЛЕЕ` | Внутри `ПОВТОРЯЙ` | `Mod.Limit` |

Аналогично, `ИСПОЛЬЗУЯ` внутри `ДУМАЙ` принимает список инструментов, а внутри `ВЫПОЛНИ` — ровно один. Это один абстрактный ID `Mod.Using`, семантика определяется оператором.

## Пример: исследовательский агент

```coil
' ──────────────────────────────────────────────
' Окружение
' ──────────────────────────────────────────────
УЧАСТНИКИ эксперт, автор
ИНСТРУМЕНТЫ поиск

' ──────────────────────────────────────────────
' Скиллы — квалификация решателя.
' Определяем, какими знаниями и подходом
' должна обладать LLM при решении задачи.
' ──────────────────────────────────────────────
ОПРЕДЕЛИ research_skill
<<
Ты — аналитик-исследователь.
Работаешь с неструктурированными запросами.
Всегда ищешь первоисточники и указываешь
степень уверенности в выводах. Избегаешь спекуляций.
Если данных недостаточно — говоришь об этом прямо.
>>
КОНЕЦ

ОПРЕДЕЛИ domain_expertise
<<
Ты разбираешься в продуктовой аналитике.
Знаешь метрики retention, activation, churn.
Умеешь читать воронки и когортные отчёты.
При анализе всегда разделяешь корреляцию
и причинно-следственную связь.
>>
КОНЕЦ

' ──────────────────────────────────────────────
' Входные данные — среда обязана предоставить
' значения, прежде чем протокол продолжится.
' ──────────────────────────────────────────────
ПОЛУЧИ запрос
<<
Запрос пользователя для исследования.
>>
КОНЕЦ

ПОЛУЧИ история_обращений
<<
История предыдущих обращений по этой теме.
>>
КОНЕЦ

ПОЛУЧИ кейс
<<
Идентификатор текущего кейса.
>>
КОНЕЦ

' ──────────────────────────────────────────────
' 1. Постановка задачи — как в математике:
'    КАК      — квалификация решателя
'    ЦЕЛЬ     — зачем решаем
'    ВХОД     — условие задачи
'    КОНТЕКСТ — дополнительные данные
'    РЕЗУЛЬТАТ — что найти
'
' ДУМАЙ запускается в фоне. Создаёт обещание
' ?анализ и поток ~анализ.
' Условия можно дополнять во время решения
' через СИГНАЛ.
' ──────────────────────────────────────────────
ДУМАЙ анализ
  КАК $research_skill, $domain_expertise
  ЦЕЛЬ <<
  Подготовить обоснованный ответ с источниками.
  >>
  ВХОД <<
  Исследуй запрос пользователя. Учитывай все данные,
  которые поступят в процессе через поток.
  >>
  КОНТЕКСТ <<
  Запрос: $запрос
  История обращений: $история_обращений
  >>
  РЕЗУЛЬТАТ
  * ответ: ТЕКСТ - итоговый ответ
  * источники: СПИСОК - использованные материалы
    * ref: ТЕКСТ - ссылка
    * why: ТЕКСТ - почему релевантен
  * уверенность: ВЫБОР(высокая, средняя, низкая)
КОНЕЦ

' ──────────────────────────────────────────────
' 2. Сбор данных — параллельно с размышлением.
'    Поиск и запрос эксперту не блокируют
'    ни друг друга, ни ДУМАЙ.
' ──────────────────────────────────────────────
ВЫПОЛНИ найденное
  ИСПОЛЬЗУЯ !поиск
  - query: $запрос
  - limit: 10
КОНЕЦ

НАПИШИ мнение
  КУДА #консультации
  КОМУ @эксперт
  ЖДАТЬ ЛЮБОЙ
  НЕ БОЛЕЕ 10м
  <<
  Нужна экспертиза по запросу: $запрос
  >>
КОНЕЦ

' ──────────────────────────────────────────────
' 3. Дополнение условий во время решения.
'    Данные приходят в произвольном порядке —
'    каждый результат сразу попадает в ~анализ.
'    LLM учитывает их без перезапуска.
' ──────────────────────────────────────────────
ПОВТОРЯЙ 2
  ЖДИ данные
    НА ?найденное, ?мнение
    РЕЖИМ ЛЮБОЙ
  КОНЕЦ

  СИГНАЛ ~анализ
    <<
    $данные
    >>
  КОНЕЦ
КОНЕЦ

' ──────────────────────────────────────────────
' 4. Решение готово.
'    Анализ учёл исходные данные,
'    результаты поиска и мнение эксперта.
' ──────────────────────────────────────────────
ЖДИ
  НА ?анализ
КОНЕЦ

' ──────────────────────────────────────────────
' 5. Ответ автору запроса.
' ──────────────────────────────────────────────
НАПИШИ
  КУДА #результаты/$кейс
  КОМУ @автор
  <<
  $анализ.ответ
  >>
КОНЕЦ

ВЫХОД
```

## Заметки

**Почему НАПИШИ, а не ОТПРАВЬ?**
Пользовательский ментальный фрейм: «напиши в канал», «напиши эксперту». Это слово, которое используют в мессенджерах. `ОТПРАВЬ` звучит технически — как будто ты работаешь с почтовым сервером, а не пишешь коллеге.

**Почему ПОЛУЧИ, а не ВОЗЬМИ?**
`ПОЛУЧИ` — блокирующая привязка: протокол стоит, пока среда не предоставит значение. «Получить» подразумевает ожидание — тебе дают, а не ты забираешь. `ВОЗЬМИ` предполагает агентивность, которой у протокола на этом шаге нет.

**Почему КУДА и КОМУ, а не В и ДЛЯ?**
`КУДА` — адрес канала (куда доставить). `КОМУ` — адресат (кому предназначено). Различие между направлением и получателем зеркалит два уровня адресации: канал и участник. `В` и `ДЛЯ` были бы слишком короткими и многозначными.

**Почему НЕ БОЛЕЕ, а не ТАЙМАУТ / ЛИМИТ?**
Русский язык позволяет использовать одну естественную фразу для обоих случаев: «не более 10 минут» (таймаут) и «не более 5 раз» (лимит итераций). Контекст оператора однозначно определяет, о чём речь. Заимствования `ТАЙМАУТ` и `ЛИМИТ` были бы понятны, но нарушили бы стилистику.

## FAQ

**Q: Это «настоящий» COIL?**
A: В COIL нет диалекта по умолчанию. Русский стандартный и английский стандартный — оба часть официальной поставки. Они маппят ключевые слова на одни и те же абстрактные операторы и дают идентичное поведение в рантайме. Спецификация языка использует русские ключевые слова, но это не делает русский привилегированным — это выбор автора спеки.

**Q: Можно смешивать языки в одном скрипте?**
A: Нет. Скрипт пишется на одном диалекте. Сигилы (`$`, `?`, `@`, `!`, `#`, `~`) универсальны, но ключевые слова должны быть из одного диалекта.

**Q: Идентификаторы — тоже по-русски?**
A: Идентификаторы (имена переменных, участников, инструментов) свободные. Можно писать `DEFINE мой_результат` в английском диалекте и `ОПРЕДЕЛИ my_result` в русском. Диалект определяет ключевые слова, а не идентификаторы.

**Q: Другие диалекты?**
A: COIL поддерживает community-диалекты для любого языка и тематики. См. также: [English standard](../en-standard/README.md), [English profanity](../en-profanity/README.md), [Matrix theme](../ru-matrix/README.md), [Русский мат](../ru-mat/README.md).
