# Спецификация диалектной таблицы

| | |
|---|---|
| **Статус** | Черновик |
| **Назначение** | Формат диалектной таблицы, полный реестр абстрактных идентификаторов, правила составления диалекта |

---

## 1. Метаданные

| Поле | Обязательность | Описание |
|---|---|---|
| `name` | Да | Машинное имя диалекта: `en-standard`, `ru-matrix` |
| `label` | Да | Человекочитаемое название: `Standard English`, `Красная таблетка` |

---

## 2. Категории маппинга

Диалектная таблица обязана содержать маппинг для всех абстрактных идентификаторов из шести категорий:

| Категория | Префикс | Что покрывает |
|---|---|---|
| Операторы | `Op.*` | Команды верхнего уровня |
| Терминаторы | `Kw.*` | Закрытие блока |
| Модификаторы | `Mod.*` | Именованные части операторов |
| Политики | `Pol.*` | Enum-значения внутри модификаторов |
| Типы результата | `Typ.*` | Типы в микросинтаксисе `РЕЗУЛЬТАТ` |
| Суффиксы длительности | `Dur.*` | Единицы duration literals |

---

## 3. Правила

1. Каждый абстрактный идентификатор из реестра должен иметь ровно одну ключевую фразу в диалекте.
2. Одна ключевая фраза может соответствовать нескольким абстрактным идентификаторам, если они однозначно разрешаются по контексту оператора.
3. Ключевая фраза — одно или несколько слов. Регистр значим.
4. Ключевая фраза не должна начинаться с сигила (`$`, `?`, `@`, `!`, `#`, `~`) или цифры.
5. Ключевая фраза не должна конфликтовать с синтаксическими элементами ядра: `<<`, `>>`, `*`, `-`, `'`, `:`.
6. Абстрактные идентификаторы используют английские мнемоники как конвенцию кода реализации. Это не привилегированный диалект — пользователь никогда не видит абстрактные идентификаторы.

---

## 4. Полный реестр абстрактных идентификаторов

### 4.1. Операторы (Core)

| ID | Категория | Описание |
|---|---|---|
| `Op.Actors` | Мгновенный | Объявление участников |
| `Op.Tools` | Мгновенный | Объявление инструментов |
| `Op.Define` | Мгновенный | Создание именованного значения |
| `Op.Set` | Мгновенный | Изменение существующего значения |
| `Op.Receive` | Блокирующий | Привязка значения из хост-среды |
| `Op.Think` | Запускающий | Когнитивный шаг LLM |
| `Op.Execute` | Запускающий | Вызов инструмента |
| `Op.Send` | Запускающий | Отправка сообщения |
| `Op.Wait` | Блокирующий | Точка синхронизации |
| `Op.Exit` | Завершающий | Завершение протокола |

### 4.2. Операторы (Extended)

| ID | Категория | Описание |
|---|---|---|
| `Op.If` | Control-flow | Условное ветвление |
| `Op.Repeat` | Control-flow | Цикл с обязательным лимитом |
| `Op.Each` | Control-flow | Итерация по списку |
| `Op.Gather` | Control-flow | Агрегация результатов |
| `Op.Signal` | Запускающий | Отправка данных в существующий поток |

### 4.3. Терминаторы

| ID | Описание |
|---|---|
| `Kw.End` | Закрытие блока |

### 4.4. Модификаторы — оснащение (`Op.Think`)

| ID | Вид | Описание |
|---|---|---|
| `Mod.Via` | Ссылочный | Модель LLM |
| `Mod.As` | Ссылочный | Квалификация решателя (скиллы) |
| `Mod.Using` | Ссылочный | Доступные инструменты (в `Op.Think` — список; в `Op.Execute` — ровно один) |

### 4.5. Модификаторы — постановка (`Op.Think`)

| ID | Вид | Описание |
|---|---|---|
| `Mod.Goal` | Шаблонный | Цель когнитивного шага |
| `Mod.Input` | Шаблонный | Условие задачи |
| `Mod.Context` | Шаблонный | Дополнительные данные |
| `Mod.Result` | Структурный | Что LLM должна определить |

### 4.6. Модификаторы — адресация (`Op.Send`)

| ID | Вид | Описание |
|---|---|---|
| `Mod.To` | Ссылочный | Адрес канала |
| `Mod.For` | Ссылочный | Получатели |
| `Mod.ReplyTo` | Ссылочный | Ссылка на сообщение-источник |
| `Mod.Await` | Режимный | Политика ожидания ответа |
| `Mod.Timeout` | Ссылочный | Таймаут ожидания (также используется в `Op.Wait`) |

### 4.7. Модификаторы — синхронизация (`Op.Wait`)

| ID | Вид | Описание |
|---|---|---|
| `Mod.On` | Ссылочный | Ожидаемые обещания |
| `Mod.Mode` | Режимный | Режим ожидания |

### 4.8. Модификаторы — итерация (`Op.Repeat`, `Op.Each`)

| ID | Вид | Описание |
|---|---|---|
| `Mod.Until` | Ссылочный | Условие выхода из цикла |
| `Mod.Limit` | Ссылочный | Лимит итераций (обязателен в `Op.Repeat`) |
| `Mod.From` | Ссылочный | Источник списка для `Op.Each` |

### 4.9. Политики

| ID | Описание |
|---|---|
| `Pol.None` | Не ждать ответа |
| `Pol.Any` | Достаточно первого |
| `Pol.All` | Нужны все |

### 4.10. Типы результата

| ID | Описание |
|---|---|
| `Typ.Text` | Строковое значение |
| `Typ.Number` | Числовое значение |
| `Typ.Flag` | Булево значение |
| `Typ.Choice` | Одно из перечисленных значений |
| `Typ.List` | Массив элементов со структурой |

### 4.11. Суффиксы длительности

| ID | Описание |
|---|---|
| `Dur.Seconds` | Секунды |
| `Dur.Minutes` | Минуты |
| `Dur.Hours` | Часы |

## 5. Контекстное разрешение

Одна ключевая фраза может маппиться на несколько абстрактных идентификаторов, если контекст оператора однозначно определяет, какой идентификатор имеется в виду.

Известные случаи:

| Ключевая фраза (RU) | Контекст | Абстрактный ID |
|---|---|---|
| `НЕ БОЛЕЕ` | Внутри `Op.Send` или `Op.Wait` | `Mod.Timeout` |
| `НЕ БОЛЕЕ` | Внутри `Op.Repeat` | `Mod.Limit` |
| `ИСПОЛЬЗУЯ` | Внутри `Op.Think` | `Mod.Using` (список, необязательный) |
| `ИСПОЛЬЗУЯ` | Внутри `Op.Execute` | `Mod.Using` (один, обязательный) |

В последнем случае это один и тот же абстрактный идентификатор `Mod.Using` — различие в семантике определяется оператором.

Обратный маппинг (для диагностики) должен учитывать контекст оператора при выборе ключевой фразы.
