# Решения

Журнал принятых проектных решений по языку COIL.

## Как ведётся этот журнал

Каждое проектное решение получает сквозной номер `D-NNNN`.

| Статус | Значение |
|---|---|
| `принят` | Решение зафиксировано и отражено в spec без серьёзных незакрытых дыр. |
| `принят как направление` | Архитектурный выбор сделан, но точная форма ещё уточняется в spec. |
| `заменён D-NNNN` | Решение заменено указанным. |

Процесс:
1. Принять решение → присвоить следующий `D-NNNN`, записать мотивацию, статус `принят`.
2. Заменить решение → старому присвоить `заменён D-NNNN`, новому — свой `D-NNNN`.

> Открытые вопросы живут в GitHub Issues, а не в этом журнале.

## D-0028 — `ВЫЗОВИ` объединён с `ВЫПОЛНИ`

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-20 |
| **Влияет на** | `spec/02-core.md`, `spec/04-execution-model.md`, `spec/10-terminology.md` |
| **Контекст** | Различие one-shot vs session не наблюдаемо для пользователя на уровне намерения. Пользователь хочет «использовать инструмент», а режим — свойство инструмента, а не выбор оператора. |
| **Решение** | `ВЫПОЛНИ` — единственный оператор работы с инструментами в Core. `ВЫЗОВИ` переходит в устаревшие формы. Аргументы — только `- key: value`; шаблонное тело `<< >>` запрещено. `ВЫПОЛНИ` всегда создаёт `?name` и `~name`; для one-shot инструмента поток сразу закрывается. |
| **Последствия** | Имя `ВЫПОЛНИ` семантически шире и естественнее; разделение `<< >>` (естественный язык) vs `- key: value` (машинные вызовы) становится принципиальным. Заменяет D-0014. |

## D-0027 — Трёхуровневая иерархия субъектов спецификации

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-17 |
| **Влияет на** | `spec/10-terminology.md`, `spec/04-execution-model.md`, `spec/09-os-integration.md` |
| **Контекст** | Единый «рантайм ОС» размывал ответственность: неясно, кто валидирует синтаксис, кто исполняет шаги, кто поставляет участников. |
| **Решение** | Три субъекта: **Реализация COIL** (синтаксис и валидность), **Среда исполнения** (шаги, promises, потоки, budget), **Хост-среда** (участники, инструменты, адреса, события). Термин «рантайм» допустим только в пояснительных текстах. |
| **Последствия** | Одна конструкция может порождать обязательства на нескольких уровнях (например, `РЕЗУЛЬТАТ` валидируется реализацией при подготовке и средой исполнения при получении когнитивного ответа). |

## D-0026 — Две группы модификаторов `ДУМАЙ`

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-17 |
| **Влияет на** | `spec/02-core.md` |
| **Контекст** | Модификаторы `ДУМАЙ` различаются по природе: одни определяют ресурсы шага, другие — саму задачу. |
| **Решение** | Оснащение (`ЧЕРЕЗ`, `КАК`, `ИСПОЛЬЗУЯ`) определяет, как исполняется шаг. Постановка (`ЦЕЛЬ`, `ВХОД`, `КОНТЕКСТ`, `РЕЗУЛЬТАТ`) определяет, что решать. Оснащение предшествует постановке. |
| **Последствия** | Все модификаторы оснащения необязательны. Порядок становится проверяемым на этапе валидации. |

## D-0025 — `ИСПОЛЬЗУЯ` в `ДУМАЙ`

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-17 |
| **Влияет на** | `spec/02-core.md` |
| **Контекст** | LLM в ходе размышления может нуждаться в инструментах, но вызов должен быть по её усмотрению, а не обязательным. |
| **Решение** | Модификатор `ИСПОЛЬЗУЯ !tool1, !tool2` предоставляет LLM набор доступных инструментов. В отличие от `ИСПОЛЬЗУЯ` в `ВЫПОЛНИ` (один обязательный инструмент), здесь — список доступных, ноль или более вызовов каждого. |
| **Последствия** | Все инструменты должны быть объявлены через `ИНСТРУМЕНТЫ`. |

## D-0024 — `ЧЕРЕЗ` — выбор модели

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-17 |
| **Влияет на** | `spec/02-core.md` |
| **Контекст** | Скрипт должен указывать, какую модель использовать, но без хардкода имени модели. |
| **Решение** | Модификатор `ЧЕРЕЗ $name` принимает только ссылку, не литерал — модель определяется через `ОПРЕДЕЛИ` или `ПОЛУЧИ`. Если опущен, среда исполнения использует модель по умолчанию. |
| **Последствия** | Скрипт отделён от конфигурации. |

## D-0023 — Аргументы инструментов `- key: value`

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-17 |
| **Влияет на** | `spec/02-core.md` |
| **Контекст** | Инструменты принимают структурированные данные, а не естественный язык. |
| **Решение** | Формат `- key: value`, значения — литералы или ссылки `$name`. Шаблонное тело `<< >>` в `ВЫПОЛНИ` запрещено. |
| **Последствия** | Два мира: `<< >>` — для естественного языка (`ДУМАЙ`, `НАПИШИ`), `- key: value` — для машинных вызовов (`ВЫПОЛНИ`). |

## D-0022 — Структура `ДУМАЙ` — постановка задачи

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-17 |
| **Влияет на** | `spec/02-core.md` |
| **Контекст** | `ДУМАЙ` должен формализовать задачу для LLM, а не быть свободной формой. |

## D-0021 — Компактная форма `<<`

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-17 |
| **Влияет на** | `spec/06-templates.md`, `spec/02-core.md` |
| **Контекст** | `<<` на отдельной строке создавал визуальный разрыв между именем модификатора и его содержимым. |
| **Решение** | `<<` пишется на той же строке, что и имя модификатора: `ЦЕЛЬ <<`. Развёрнутая форма остаётся валидной, но в примерах используется компактная. |
| **Последствия** | Единообразие примеров в спецификации. |

## D-0020 — `КАК` — ссылочный модификатор

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-17 |
| **Влияет на** | `spec/02-core.md` |
| **Контекст** | `КАК` мог бы принимать шаблон `<< >>`, но скилл — это именованная сущность, а не свободный текст. |
| **Решение** | `КАК` принимает список ссылок на скиллы через запятую (`КАК $role, $skill`), а не шаблон. |
| **Последствия** | Скилл — именованный блок знаний/поведения; допускается перечисление. |

## D-0019 — Inline-декларации без `КОНЕЦ`

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-17 |
| **Влияет на** | `spec/02-core.md` |
| **Контекст** | Операторы, чьё содержимое — список имён, а не тело, не нуждаются в закрывающем `КОНЕЦ`. |
| **Решение** | Inline-форма: `УЧАСТНИКИ тех_аналитик, клиент_а` — одна строка, без `КОНЕЦ`. Блочная форма с `КОНЕЦ` тоже валидна. Применимо к `УЧАСТНИКИ`, `ИНСТРУМЕНТЫ`. |
| **Последствия** | Тело оператора не выносится в сигнатуру. `ОПРЕДЕЛИ` остаётся блочным. `ВЫХОД` пишется одной строкой без `КОНЕЦ`. |

## D-0018 — Два представления: COIL-C и COIL-H

| | |
|---|---|
| **Статус** | принят |
| **Решено** | 2026-03-17 |
| **Влияет на** | `spec/00-overview.md`, `spec/11-coil-h.md` |
| **Контекст** | Человеку нужна читаемость, LLM — однозначность формы. Вместо компромисса — две проекции. |
| **Решение** | **COIL-C** (canonical) — машинная форма. **COIL-H** (human) — читаемая проекция. COIL-H транслируется в COIL-C без потерь. Семантика живёт в COIL-C. |
| **Последствия** | COIL-H не имеет собственной семантики; редактор обязан транслировать COIL-H в COIL-C. |
