# 04. Execution model

| | |
|---|---|
| **Статус** | Черновик |
| **Назначение** | Модель исполнения протокола: порядок шагов, детерминизм, обещания |

---

## 4.1. Что исполняет COIL

COIL описывает **протокол поведения агента**. Он не описывает вечный цикл жизни агента и не заменяет scheduler хост-среды.

### Разделение уровней

| Уровень | Ответственность |
|---|---|
| **Хост-среда** | Жизненный цикл агента в рамках сервера: приём событий и решение о запуске нового протокола |
| **COIL-протокол** | Что делать внутри конкретного запуска |

---

## 4.2. Категории шагов

### По роли в протоколе

| Роль | Операторы |
|---|---|
| Декларации окружения | `УЧАСТНИКИ`, `ИНСТРУМЕНТЫ` |
| Связывание значения | `ОПРЕДЕЛИ`, `ПОЛУЧИ` |
| Изменение значения | `УСТАНОВИ` |
| Запуск действия | `ДУМАЙ`, `ВЫПОЛНИ`, `НАПИШИ` |
| Ожидание | `ЖДИ` |
| Завершение | `ВЫХОД` |

### По наблюдаемому поведению

| Поведение | Операторы |
|---|---|
| Мгновенные | `УЧАСТНИКИ`, `ИНСТРУМЕНТЫ`, `ОПРЕДЕЛИ`, `УСТАНОВИ` |
| Блокирующие | `ПОЛУЧИ`, `ЖДИ` |
| Запускающие асинхронную работу | `ДУМАЙ`, `ВЫПОЛНИ`, `НАПИШИ` |
| Завершающие | `ВЫХОД` |

---

## 4.3. Детерминизм

Среда исполнения обязана обеспечивать детерминированно:

- разбор синтаксиса;
- декларации и связывание значений;
- порядок шагов;
- адресацию;
- ожидание;
- таймауты;
- budget-checks;
- вычисление условий и циклов;
- маршрутизацию событий (см. [07-streams.md § 7.8](07-streams.md));
- валидацию результата.

Недетерминированы по природе источника:

- результат `ДУМАЙ`;
- ответы других участников;
- результаты инструментов;
- внешние события хост-среды.

---

## 4.4. Обещания и значения

Запускающий оператор создаёт обещание результата `?name`.

После успешного ожидания итоговое значение доступно как `$name`.

```coil
ДУМАЙ plan
...
КОНЕЦ

ЖДИ
НА ?plan
КОНЕЦ

НАПИШИ
КУДА #case/123
<<
Итог: $plan.summary
>>
КОНЕЦ
```

---

## 4.5. Порядок исполнения

Внутри одного протокола шаги исполняются последовательно, если не запускают отдельную асинхронную работу.

Пример порядка:

1. `ДУМАЙ plan` — запуск когнитивной работы, протокол идёт дальше.
2. `НАПИШИ answer` — запуск коммуникации, протокол идёт дальше.
3. `ЖДИ` — явная точка синхронизации.

---

## 4.6. `ПОЛУЧИ` не замораживает весь агент

`ПОЛУЧИ` блокирует только **текущий экземпляр протокола**. Это не означает остановку всего агента в хост-среде.

> Примечание: это важно для модели, в которой один агент должен уметь одновременно ждать данные в одном экземпляре протокола и начинать новый по другому mention.

---

## 4.7. Политика ожидания у `НАПИШИ`

`ЖДАТЬ` является свойством **конкретного сообщения**, а не универсальным параметром `ЖДИ`.

Именно `НАПИШИ` определяет, когда обещание `?name` считается завершённым:

| Политика | Условие завершения |
|---|---|
| `ЖДАТЬ НЕТ` | Сообщение отправлено |
| `ЖДАТЬ ЛЮБОЙ` | Получен хотя бы один ответ |
| `ЖДАТЬ ВСЕ` | Получены ответы от всех адресатов |

`ЖДИ` лишь приостанавливает протокол до нужного момента.

---

## 4.8. Несколько протоколов одного агента

Хост-среда обязана поддерживать несколько независимых экземпляров протокола для одного и того же агента:

- по разным корневым сообщениям;
- по разным цепочкам комментариев.

Это свойство хост-среды, а не отдельная конструкция COIL.
