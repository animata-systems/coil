# 07. Streams

| | |
|---|---|
| **Статус** | Смешанный (см. § 7.1a) |
| **Назначение** | Потоки, события, сигналы |

---

## 7.1. Зачем потоки нужны языку

Простая модель «запусти и дождись финала» недостаточна для агентной ОС. Реальные сценарии требуют:

- запустить размышление LLM;
- параллельно написать тех. отделу;
- открыть внешний диалог с клиентом;
- по мере поступления новых данных отправлять уточнения в уже идущее размышление.

Это приводит к модели **открытых потоков / сессий**.

---

## 7.1a. Статусная карта

| Область | Статус |
|---|---|
| Пространство имён `~name` зарезервировано за stream references | Stable |
| Потоки как отдельный класс сущностей, отличный от `?promise` и `$value` | Stable |
| Какие операторы гарантированно создают `~name` | Experimental |
| Событийная форма `ЖДИ ... ИЗ ~...` | Experimental |
| `СИГНАЛ ~name` | Experimental |
| Shape event object | Experimental |
| Момент и модель доставки сигнала в поток | Experimental |
| Ownership потока, buffering, broadcast vs compete | Deferred |
| Ordering guarantees, end-of-stream / close semantics | Deferred |
| Cancellation / interruption model | Deferred |

---

## 7.2. Поток и обещание

Запускающий оператор создаёт:

| Сущность | Ссылка | Назначение |
|---|---|---|
| Обещание финального результата | `?name` | Терминальная проекция |
| Открытый поток | `~name` | Живая двусторонняя сессия |

Примеры:

| Оператор | Обещание | Поток |
|---|---|---|
| `ДУМАЙ решение` | `?решение` | `~решение` |
| `НАПИШИ ответ` | `?ответ` | `~ответ` |
| `ВЫПОЛНИ диалог` | `?диалог` | `~диалог` |

---

## 7.3. События и сигналы

Поток обладает двумя направлениями взаимодействия:

| Направление | Механизм | Описание |
|---|---|---|
| Из потока | **Событие** | Данные, поступающие из открытого потока |
| В поток | **Сигнал** | Данные, отправляемые оператором `СИГНАЛ` |

> Примечание: термин «interrupt» сознательно не используется. В агентной ОС модель мягче: сигнал ставится в очередь потока, поток принимает его в согласованной точке исполнения. Обработка кооперативная, а не hard-preemptive.

---

## 7.4. `ЖДИ` для событий

> Статус: experimental.

Кроме ожидания обещаний, `ЖДИ` должен уметь ждать **следующее событие** из потоков.

### Форма (experimental)

```coil
ЖДИ event
ИЗ ~решение, ~ответ, ~диалог
НЕ БОЛЕЕ 10м
КОНЕЦ
```

### Семантика

- `$event` — объект события, по которому можно ветвиться.
- Минимально ожидаемые поля события: источник, тип, данные, время.

> Примечание: точный shape события не зафиксирован.

---

## 7.5. `СИГНАЛ`

> Статус: experimental.

```coil
СИГНАЛ ~решение
<<
Клиент сообщил новый номер договора: $contract_id
>>
КОНЕЦ
```

`СИГНАЛ` не создаёт новый поток. Он продолжает уже существующий.

---

## 7.6. `ВЫПОЛНИ`

`ВЫПОЛНИ` — единый оператор работы с инструментами. В потоковой модели он создаёт `?name` и `~name`. Для one-shot инструментов поток закрывается сразу; для интерактивных сессий — остаётся открытым.

```coil
ВЫПОЛНИ диалог
ИСПОЛЬЗУЯ !открыть_чат_с_клиентом
- topic: "уточнение номера договора и времени сбоя"
КОНЕЦ
```

---

## 7.7. Типовой сценарий с потоками

```coil
ДУМАЙ решение
...
КОНЕЦ

НАПИШИ тех
КУДА #быстрые_вопросы
КОМУ @тех_отдел
ЖДАТЬ ЛЮБОЙ
<<
Нужна возможная причина сбоя и временный обход.
>>
КОНЕЦ

ВЫПОЛНИ диалог
ИСПОЛЬЗУЯ !открыть_чат_с_клиентом
- topic: "уточнение номера договора и времени сбоя"
КОНЕЦ
```

После запуска среда исполнения обязана ждать события и по мере их поступления доставлять сигналы в `~решение`.

---

## 7.8. Маршрутизация событий

> Статус: deferred.

Манифест относит маршрутизацию событий к жёсткой части, исполняемой средой исполнения детерминированно. Финальная модель сознательно не нормализуется в текущем цикле. Следующие вопросы остаются открытыми:

| Вопрос | Описание |
|---|---|
| **Доставка экземпляру** | По каким правилам среда исполнения должна доставлять событие конкретному экземпляру протокола? |
| **Событие без слушателя** | Что происходит с событием, которое никто не ждёт: теряется, буферизируется, порождает ошибку? |
| **Конкуренция за событие** | Может ли несколько экземпляров ждать событие из одного потока? Кто получает: все (broadcast) или один (compete)? |
| **Порядок доставки** | Должна ли среда исполнения гарантировать порядок доставки событий внутри одного потока? |

> Рабочая гипотеза: поток принадлежит экземпляру протокола, создавшему его. События доставляются только этому экземпляру. Если экземпляр не ждёт — события буферизируются до следующего `ЖДИ` или до таймаута.

---

## 7.9. Что считается архитектурно принятым

- Открытые потоки нужны языку.
- Размышление LLM — это не только «запуск и финал», но и живой канал для новых данных.
- Интерактивные инструменты с двусторонней связью укладываются в общую потоковую модель.
