# 05. Structured output

| | |
|---|---|
| **Статус** | Черновик |
| **Назначение** | Спецификация модификатора `РЕЗУЛЬТАТ`: микросинтаксис, типы, валидация |

---

## 5.1. Базовый принцип

`РЕЗУЛЬТАТ` — это **декларация того, что LLM должна определить**, а не описание формата вывода и не постфактумная JSON-схема.

COIL использует `РЕЗУЛЬТАТ` как **единственный источник истины**: одновременно спецификацию когнитивной задачи и машинную схему результата. Отдельное дублирование prose-инструкции и schema запрещено.

---

## 5.2. Расположение

`РЕЗУЛЬТАТ` — структурный модификатор оператора `ДУМАЙ`.

Правила:

- `РЕЗУЛЬТАТ` должен идти **последним** среди модификаторов.
- `РЕЗУЛЬТАТ` не закрывается собственным `КОНЕЦ`.
- Блок завершается общим `КОНЕЦ` оператора `ДУМАЙ`.

> Примечание: это решение принято, чтобы `РЕЗУЛЬТАТ` не выглядел как отдельный оператор.

---

## 5.3. Микросинтаксис

Каждая строка верхнего уровня имеет форму:

```
* [имя поля]: [тип] - [описание]
```

Пример:

```coil
РЕЗУЛЬТАТ
* summary: ТЕКСТ - краткий общий вывод
* artifacts: СПИСОК - список артефактов, требующих изменений
  * path: ТЕКСТ - путь к артефакту
  * action: ВЫБОР(создание, изменение, удаление) - тип изменения
  * reason: ТЕКСТ - почему требуется изменение
```

---

## 5.4. Типы

| Тип | Назначение |
|---|---|
| `ТЕКСТ` | Строковое значение |
| `ЧИСЛО` | Числовое значение |
| `ФЛАГ` | Булево значение |
| `ВЫБОР(...)` | Одно из перечисленных значений |
| `СПИСОК` | Массив элементов со структурой |

> Предупреждение: синонимы (`TEXT`, `STRING`, `BOOL`, `ENUM` и т.д.) запрещены в исходном языке. Реализация COIL не должна принимать их как валидные.

---

## 5.5. Вложенность

### Разрешено

- Корневая запись с простыми полями.
- Поля типа `СПИСОК`.
- Под `СПИСОК` — структура **одного элемента списка**.

### Запрещено в Core

- Произвольные деревья.
- Вложенные списки.
- `ОБЪЕКТ` внутри `ОБЪЕКТ`.
- Рекурсивные структуры.
- Union-типы общего вида.

> Примечание: это компромисс между выразительностью и простотой для человека и LLM.

---

## 5.6. Компиляция в машинную схему

Исходный текст `РЕЗУЛЬТАТА` не является JSON и не обязан быть JSON Schema.

Реализация COIL вправе компилировать его в:

- внутреннюю типовую модель;
- structured schema;
- формат валидации LLM-ответа.

---

## 5.7. Repair policy

Если результат `ДУМАЙ` не соответствует `РЕЗУЛЬТАТУ`, среда исполнения обязана действовать по repair-policy:

1. Валидация результата.
2. Одна или несколько repair-попыток против **того же самого `РЕЗУЛЬТАТА`**.
3. Если repair не привёл к валидному результату — ошибка исполнения.

Repair не должен менять high-level смысл задачи. Он исправляет только несоответствие структуре.

---

## 5.8. Практический смысл описания

Часть после `-` — это не decoration, а семантическая подсказка модели.

```coil
* reason: ТЕКСТ - почему требуется изменение
```

означает не просто «верни поле reason», а «выведи причину, по которой изменение нужно».

Именно поэтому `РЕЗУЛЬТАТ` одновременно служит спецификацией когнитивной задачи и машинной схемой результата.
